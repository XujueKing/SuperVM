### 4. 数据溯源核查 ✅

**290K TPS 来源验证**:

- 出处: `BENCHMARK_RESULTS.md` §0 高竞争 TPS 实测

- 硬件: i7-9750H 6C/12T, Windows 本地

- 配置: 10线程 × 100笔/线程 = 1000笔总交易

- 冲突率: 80% (5个热键)

- 结果:
  - 无 Bloom: 3.59ms → 278,800 TPS
  - 有 Bloom: 3.46ms → 290,700 TPS

- 计算: TPS = 1000笔 / (耗时秒) = 1000 / 0.00346 ≈ 290,700

## 💡 技术要点总结

### 为什么能达到 800K–1.2M TPS？

**三大护城河**:
1. **MVCC 并行执行**: 单线程 242K TPS, >90% 扩展效率
2. **2PC 批量提交**: 并发可扩展, 流水线化
3. **RocksDB 自适应**: 峰值 860K ops/s

**数学上界**:

```

TPS ≈ min(
  T_exec(threads, conflict),    // 执行引擎
  T_commit_ops / writes_per_tx,  // 提交批量
  T_storage_qps                  // 存储上限
)

```

**工程路径**:

- 并行执行 (MVCC): 读写分离, 冲突延迟检测

- 批量流水线: 锁获取与版本追加解耦

- 并行校验 (Rayon): 读集合并行验证

- 自适应批量 (8-128): 动态调整批次

- 存储自适应: chunk 找到最优平衡

每环节 1.2×~3.4× 增益 → 叠乘形成数量级提升
